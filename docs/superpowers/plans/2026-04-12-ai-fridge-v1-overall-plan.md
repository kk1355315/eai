# AI Fridge V1 Overall Plan

## 这项目到底要做什么

这个项目不是单纯做一个“识别食物图片”的模型。

V1 的目标是做一个能在冰箱场景里工作的最小可用版本：

1. 先识别冰箱里放进去的主要食材是什么
2. 记录这个食材是什么时候放进去的
3. 去查一套食品安全建议
4. 结合时间给出保守提醒

最终输出不是“这看起来像苹果”这么简单，而是：

- 这是什么食材
- 已经放了多久
- 现在还建议吃 / 尽快吃 / 不建议吃

## V1 边界

第一版只做这些：

- 食材识别
- 放入和取出时间记录
- 保守的食品安全建议
- 简单明确的提醒结果

第一版不做这些：

- 熟食识别
- 菜名识别
- 剩菜识别
- 通过 OCR 读包装日期
- 一张图里同时识别很多东西
- 自动理解复杂烹饪状态

## V1 的核心拆分

### 1. 图像识别模块

作用：

- 看图，判断冰箱里主要是什么食材

输出：

- 一个稳定的食材类别

注意：

- 这里输出的是“图片里能稳定认出来的食材类”
- 不是 `foodkeeper.json` 里的细碎原始条目名

### 2. 时间记录模块

作用：

- 记录食材何时放入、何时取出

V1 最小需求：

- 至少有 `put_in_time`
- 有条件的话记录 `take_out_time`

### 3. 食品安全建议模块

作用：

- 根据图像识别结果，去 `foodkeeper.json` 里查真实条目
- 如果一个模型类对应多个条目，就取最保守的那条建议

原则：

- `foodkeeper.json` 是“安全建议来源”
- 不是“图像标签来源”

### 4. 提醒逻辑模块

作用：

- 结合食材类别和时间，输出用户能看懂的话

V1 建议只保留三档：

- `可以继续吃`
- `建议尽快吃`
- `不建议食用`

## V1 数据流

整条链路用人话说就是：

1. 摄像头看到冰箱里的一个主要食材
2. 模型判断它属于哪个食材类
3. 这个食材类去映射 `foodkeeper.json` 里的真实条目
4. 系统拿到对应的保守保存建议
5. 系统看这个东西已经放了多久
6. 系统给出提醒

## 为什么要先做“食材类”而不是直接照抄 FoodKeeper

因为 `foodkeeper.json` 里很多名字分得太细，图片上根本不稳定。

例如：

- 图片不一定分得出牛肉具体哪个部位
- 但能先分出“肉”、“禽”、“鱼”、“虾”

所以 V1 的正确做法是：

- 模型只认图片里稳定的大类
- 每个大类再去绑定 `foodkeeper.json` 里的真实条目

## V1 成功标准

V1 做成，至少要满足：

- 能识别一组高频冰箱食材
- 这些食材都能落回 `foodkeeper.json`
- 能根据放入时间给出保守提醒
- 提醒结果稳定、简单、能解释

## 当前工作顺序

先做：

1. 定死 V1 食材分类
2. 把每个分类映射到 `foodkeeper.json`
3. 把公开数据集准备好
4. 训练食材识别模型
5. 再接时间记录和提醒逻辑

不要倒过来做。

如果分类没定死，后面所有建议逻辑都会跟着乱。
